## 📝 子仓库提交流程

### 原则

- ✅ **每个子仓库完全独立** - 项目结构、代码风格、技术栈由项目组自行决定
- ✅ **不强制统一规范** - 根据项目特点选择合适的开发方式
- ✅ **独立的版本控制** - 每个子仓库有自己的提交历史和分支策略

### 基本工作流程

#### 1. 克隆完整项目（首次）

```bash
# 克隆主仓库及所有子模块
git clone --recursive https://github.com/YOUR_ORG/DUT-Smart-Traffic-Lab.git

# 如果已克隆但缺少子模块
git submodule update --init --recursive
```

#### 2. 在子仓库中开发

```bash
# 进入你负责的子项目
cd projects/llm-signal-optimization

# 查看当前状态
git status

# 这是一个独立的Git仓库，按常规Git流程操作即可
git checkout -b feature/your-feature
# 修改代码...
git add .
git commit -m "你的提交信息"
git push origin feature/your-feature
```

#### 3. 更新子仓库代码

```bash
# 在子仓库目录中
cd projects/llm-signal-optimization

# 拉取最新代码
git pull origin main

# 如果使用分支开发
git checkout main
git pull
git checkout your-branch
git merge main
```

#### 4. 更新主仓库的子模块引用（可选）

当子仓库有重要更新需要在主仓库记录时：

```bash
# 回到主仓库根目录
cd ../..

# 主仓库会检测到子模块的commit变化
git status
# 显示：modified:   projects/llm-signal-optimization (new commits)

# 更新主仓库对子模块的引用
git add projects/llm-signal-optimization
git commit -m "Update llm-signal-optimization to latest version"
git push
```

**注意**：这一步不是必须的。只有当你希望主仓库"锁定"子项目的特定版本时才需要执行。

## 🔄 常见操作

### 获取所有子模块的最新代码

```bash
# 方法1: 在主仓库根目录
git submodule update --remote --merge

# 方法2: 使用提供的脚本
./scripts/update-all.sh
```

### 检查所有子模块状态

```bash
# 查看所有子模块的状态
git submodule status

# 或使用脚本查看详细信息
./scripts/check-status.sh
```

### 在所有子模块中执行相同命令

```bash
# 例如：在所有子模块中切换到main分支
git submodule foreach 'git checkout main'

# 例如：在所有子模块中拉取最新代码
git submodule foreach 'git pull'
```

## 📂 子仓库创建与添加

### 新建子仓库并添加到主仓库

```bash
# 1. 在GitHub上创建新的私有仓库
#    例如：new-project

# 2. 在本地初始化并推送
mkdir new-project
cd new-project
git init
# 创建项目文件...
git add .
git commit -m "Initial commit"
git remote add origin https://github.com/YOUR_ORG/new-project.git
git push -u origin main

# 3. 回到主仓库，添加为submodule
cd /path/to/DUT-Smart-Traffic-Lab
git submodule add https://github.com/YOUR_ORG/new-project.git projects/new-project

# 4. 提交主仓库的更改
git add .gitmodules projects/new-project
git commit -m "Add new-project submodule"
git push
```

## ⚠️ 注意事项

### 1. 子模块的独立性

- 每个子仓库是**完全独立**的Git仓库
- 在子仓库中的操作不会影响主仓库（除非你主动更新引用）
- 可以有自己的分支、标签、历史记录

### 2. 提交到子仓库 vs 主仓库

```bash
# 在子仓库目录中（例如 projects/llm-signal-optimization/）
git status   # 显示子仓库的状态
git commit   # 提交到子仓库
git push     # 推送到子仓库的远程

# 在主仓库根目录（DUT-Smart-Traffic-Lab/）
git status   # 显示主仓库的状态（包括子模块引用的变化）
git commit   # 提交到主仓库
git push     # 推送到主仓库的远程
```

### 3. 避免混淆

- 始终注意你当前在哪个目录（主仓库 or 子仓库）
- 使用 `pwd` 或 `git remote -v` 确认当前仓库

### 4. 子模块不会自动更新

```bash
# ❌ 错误理解：在主仓库 git pull 会自动更新所有子模块
git pull  # 只更新主仓库，不更新子模块

# ✅ 正确做法：需要显式更新子模块
git pull
git submodule update --remote --merge
```

## 🛠️ 故障排除

### 问题1：子模块目录为空

**原因**：克隆时未使用 `--recursive` 参数

**解决**：
```bash
git submodule update --init --recursive
```

### 问题2：无法推送子模块更改

**原因**：可能在detached HEAD状态

**解决**：
```bash
cd projects/your-project
git checkout main  # 或其他分支
git pull
# 现在可以正常开发和推送
```

### 问题3：子模块显示"new commits"

**现象**：`git status` 显示子模块有新提交

**说明**：这表示子仓库的代码已更新，但主仓库的引用还指向旧版本

**处理**：
- 如果你希望主仓库跟踪最新版本：`git add projects/xxx && git commit`
- 如果不需要：忽略即可，不影响子仓库开发

### 问题4：权限错误

**错误信息**：`Permission denied` 或 `Repository not found`

**原因**：没有子仓库的访问权限

**解决**：联系管理员（chenshi_edu@gmail.com）添加权限

## 🎯 最佳实践建议

### 对于子项目开发者

1. **在子仓库中工作** - 把子仓库当作独立项目开发
2. **频繁提交推送** - 及时推送到子仓库远程，避免代码丢失
3. **使用分支开发** - 采用 feature 分支工作流（可选）
4. **定期同步** - 定期从远程拉取团队成员的更新

### 对于项目协调者

1. **定期更新主仓库引用** - 在重要里程碑时更新主仓库对子模块的引用
2. **维护文档** - 在主仓库中维护总体文档和子项目列表
3. **管理权限** - 确保团队成员有正确的访问权限
